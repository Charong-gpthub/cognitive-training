- 适用范围：认知训练平台 Web 端（9 个重点游戏 + 通用能力）

1. `/glm5/汇总/游戏测评报告.md`

## 输入结论归纳

1. 已确认问题规模：41 个（P1: 8，P2: 18，P3: 15），当前无 P0。
2. 当前测试覆盖：68%（UI + 功能）；边界测试与异常测试覆盖为 0%，是质量盲区。
3. 待办总量：58 项，预估 206h；其中账户系统单项 36h，属于高依赖任务。
4. 高频问题集中在：Flanker、Stroop、Task Switching、Hanoi 四个模块。

## 计划目标（到 2026-04-18）

1. 清零所有 P1 缺陷，确保 9 个重点游戏“可开始、可完成、可回合结束”。
2. 完成 P2 中与训练闭环直接相关的能力：进度、反馈、关键统计、必要解释。
3. 将测试覆盖从 68% 提升到 90% 以上：补齐边界/异常用例执行。
4. 完成“账号系统”最小可用方案（MVP）：第三方登录 + 云端成绩同步基础链路。

## 执行策略

1. 双线并行：`修复线`（功能与交互）+ `验证线`（对抗/回归测试）。
2. 优先顺序：先稳定性和可完成性，再做体验增强，最后做扩展功能。
3. 每周固定节奏：
- 周一：需求拆分与风险确认
- 周三：中期检查（阻塞与范围裁剪）
- 周五：回归 + 发布 + 文档更新

## 里程碑与交付

## M1（2026-03-01 ~ 2026-03-07）：P1 清零周

### 交付范围

1. Flanker：试次设置、进度显示、即时反馈、循环上限。
2. Stroop：麦克风权限拒绝提示与状态可见化。
3. Task Switching：练习模式（带提示）。
4. Hanoi：撤销功能。

### 验收标准

1. P1 未关闭数量为 0。
2. 9 个重点游戏均可完整完成一局。
3. 对抗测试方案中的 P0/P1 用例执行率 100%。

## M2（2026-03-08 ~ 2026-03-21）：P2 核心体验周

### 交付范围

1. Schulte：提示说明、非 5x5 评价标准、暂停/重置。
2. N-Back：N=0 说明增强、反应时统计、实时正确率。
3. Task Switching：试次设置、切换成本统计。
4. Iowa / Monty Hall：中期反馈、概率解释与统计可读性提升。

### 验收标准

1. 与训练结果相关的 P2 问题关闭率 >= 80%。
2. 每个重点游戏至少具备：开始入口、进度显示、结果反馈、重开路径。
3. 用户可在 3 分钟内理解并完成任一任务首局（人工抽测）。

## M3（2026-03-22 ~ 2026-04-04）：数据闭环周

### 交付范围

1. 本地存储标准化：进度保存、历史记录、训练报告汇总。
2. 统一指标字段：正确率、反应时、完成时长、有效轮次。
3. 报告页联动：支持最近 7 天/30 天趋势查看。

### 验收标准

1. 刷新/关闭后可恢复最近训练状态。
2. 报告页可展示至少 5 个任务的历史趋势。
3. 数据结构通过脱敏检查（不含本地绝对路径/隐私字段）。

## M4（2026-04-05 ~ 2026-04-18）：账号 MVP 与发布门禁周

### 交付范围

1. 第三方认证登录（GitHub）打通。
2. 云端基础同步：用户训练摘要与最近记录。
3. 发布门禁：兼容性矩阵最小集（Chrome/Edge + 移动 Chrome）通过。

### 验收标准

1. 用户可登录并看到个人历史摘要。
2. 关键流程无阻断缺陷（P0/P1 = 0，P2 可控）。
3. 回归覆盖 >= 90%，且上线前检查表全通过。

## 风险与应对

| 风险 | 影响 | 应对策略 |
|---|---|---|
| 账号系统引入后端依赖，拖慢主线修复 | 高 | 采用 MVP 范围，先 GitHub 登录 + 最小数据模型 |
| 多游戏并行改动导致回归成本激增 | 中 | 统一 UI 组件与统计字段，减少重复修复 |
| 语音相关能力受浏览器权限与兼容限制 | 中 | 提供明确降级路径（手动模式）与提示 |
| 测试执行量大，人工回归不稳定 | 中 | 固化冒烟清单，逐步脚本化关键路径 |

## 发布门禁（Go/No-Go）

2. P2 中“阻断训练闭环”的问题为 0。
3. 9 个重点游戏完成率 100%（可开局、可完成、可结算）。
4. 回归覆盖 >= 90%，并有书面结果记录。
5. 脱敏扫描通过（无本地路径/敏感配置泄露）。
